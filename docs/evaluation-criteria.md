# PydanticAI Model Factory 実装評価基準

## 概要

本ドキュメントは、PydanticAI Model Factory の比較実験における評価基準を定義します。
異なる実装アプローチ（Plan 1: 完全準拠 vs Plan 2: DRY 改善）の定量・定性評価を
統一的に行うためのフレームワークを提供します。

## 評価スコア計算方法

### 総合スコア計算式

```
総合スコア = Σ (指標スコア × 重み係数)
```

各指標について 0～100 点の正規化スコアを算出し、重み係数を乗じて合算します。

## 定量指標（合計 100%）

### 1. Cyclomatic Complexity (20%)

**定義**：
- Radon を使用したコード複雑度計測
- 関数・メソッド単位での循環的複雑度（CC）

**計測方法**：
```bash
uv run radon cc -a local_packages/image-annotator-lib/image_annotator_lib/core/pydantic_ai_factory.py
```

**評価基準**：
| CC 値 | スコア | 評価 |
|------|--------|------|
| 1～5 | 100 | 優秀（非常にシンプル） |
| 6～10 | 80 | 良好（管理可能） |
| 11～15 | 60 | 普通（複雑） |
| 16～20 | 40 | 悪い（非常に複雑） |
| 20+ | 20 | 要改善（保守困難） |

**正規化スコア**：
```
スコア = max(0, 100 - (平均CC × 5))
```

**ベンチマーク**：
- Plan 1 目標：平均 CC < 8
- Plan 2 目標：平均 CC < 6

---

### 2. コード行数 (LOC) (15%)

**定義**：
- 実装コード行数（テスト・ドキュメント除外）
- 空行・コメント行も含める

**計測方法**：
```bash
find local_packages/image-annotator-lib/image_annotator_lib/core -name "*.py" -exec wc -l {} + | tail -1
```

**評価基準**：

| LOC | スコア | 評価 |
|------|--------|------|
| 0～500 | 100 | 優秀（非常にコンパクト） |
| 501～1000 | 90 | 良好 |
| 1001～1500 | 75 | 普通 |
| 1501～2000 | 50 | やや多い |
| 2001+ | 30 | 要簡潔化 |

**正規化スコア**：
```
if LOC ≤ 500: スコア = 100
else: スコア = max(30, 100 - (LOC - 500) / 50)
```

**ベンチマーク**：
- Plan 1 目標：< 1200 LOC
- Plan 2 目標：< 1000 LOC

### 7. 新プロバイダー追加コスト (15%)

**定義**：
- 新しいプロバイダー（例：別のLLMプロバイダー）追加時の実装コスト
- 必要な変更行数・ファイル数・実装工数

**評価項目**：
1. **必要な新規ファイル数**
   - 0 ファイル：100点
   - 1 ファイル：80点
   - 2 ファイル：60点
   - 3 ファイル：40点
   - 4+ ファイル：20点

2. **必要な既存ファイル変更数**
   - 0 ファイル：100点
   - 1 ファイル：85点
   - 2 ファイル：70点
   - 3 ファイル：50点
   - 4+ ファイル：30点

3. **実装行数の増加量**
   - < 50 行：100点
   - 50～100 行：80点
   - 100～200 行：60点
   - 200～500 行：40点
   - 500+ 行：20点

**複合スコア計算**：
```
スコア = (新規ファイル数スコア × 30% +
          既存ファイル変更スコア × 30% +
          実装行数スコア × 40%)
```

**ベンチマーク**：
- Plan 1：新規ファイル 2～3、既存変更 2、追加行数 150～250
- Plan 2：新規ファイル 0～1、既存変更 1、追加行数 50～100

## 定性指標

### 1. デバッグ容易性

**評価項目**：
- エラーメッセージの明確さ
- スタックトレース情報の充実度
- 問題箇所の特定しやすさ

**評価方法**：
1. 意図的にエラーを発生させる（無効なAPIキー、無効なモデルID等）
2. エラーメッセージの明確さを 0～5 で評価
3. スタックトレースの有用性を 0～5 で評価

**計算式**：
```
スコア = (エラーメッセージ点数 + スタックトレース点数) / 10 × 100
```

**評価基準**：
| スコア | 評価 |
|--------|------|
| 90+ | 優秀：迅速に問題を特定できる |
| 70～89 | 良好：ある程度の調査で問題を特定可能 |
| 50～69 | 普通：コード読解で問題を特定可能 |
| 30～49 | やや困難：詳細な調査が必要 |
| 30未満 | 困難：特定に時間がかかる |

---

### 2. 設定の簡潔さ

**評価項目**：
- 設定ファイル（TOML）の複雑さ
- 環境変数の必要数
- 設定パラメータの直感性

**評価方法**：
1. 設定ファイル行数をカウント（コメント除外）
   - ≤ 20 行：5点
   - 21～40 行：4点
   - 41～60 行：3点
   - 61～80 行：2点
   - 80+ 行：1点

2. 必須環境変数数
   - ≤ 3：5点
   - 4～5：4点
   - 6～7：3点
   - 8～10：2点
   - 10+：1点

3. パラメータ名の直感性（5人のレビュー）
   - 4～5人が「直感的」：5点
   - 3人が「直感的」：4点
   - 2人が「直感的」：3点
   - 1人が「直感的」：2点
   - 0人が「直感的」：1点

**計算式**：
```
スコア = (設定ファイル点数 + 環境変数点数 + パラメータ直感性点数) / 15 × 100
```

---

### 3. PydanticAI 追従性

**定義**：
- PydanticAI フレームワークの最新 API への依存度
- `infer_model()` / `infer_provider()` 関数の活用度
- フレームワークアップデート時の適応性

**評価項目**：
1. **PydanticAI 最新 API の活用**
   - `infer_model()`・`infer_provider()` 使用：5点
   - 部分的に使用：3点
   - 未使用：1点

2. **手動プロバイダー判定の削減**
   - 90%+ 自動判定：5点
   - 70～89% 自動判定：4点
   - 50～69% 自動判定：3点
   - 30～49% 自動判定：2点
   - 30%未満：1点

3. **将来のフレームワークアップデートへの対応性**
   - API 変更への影響が小さい：5点
   - 中程度の影響：3点
   - 大きな影響：1点

**計算式**：
```
スコア = (API活用点数 + 自動判定点数 + 適応性点数) / 15 × 100
```

---

### 4. LoRAIro 統合影響

**定義**：
- LoRAIro メインアプリケーションとの統合時の影響
- 既存 API との互換性
- 新機能追加時の実装工数

**評価項目**：
1. **既存 API との互換性**
   - 完全互換（変更なし）：5点
   - ほぼ互換（小修正のみ）：4点
   - 部分互換（いくつかの変更）：3点
   - 互換性なし（大幅修正）：1点

2. **統合テストの作成コスト**
   - 既存テストで十分：5点
   - 小修正のみ：4点
   - 中程度の追加テスト：3点
   - 大幅な新テスト作成：1点

3. **ドキュメント更新の影響**
   - ドキュメント変更なし：5点
   - 軽微な更新：4点
   - 中程度の更新：3点
   - 大幅な更新：1点

**計算式**：
```
スコア = (互換性点数 + テスト作成点数 + ドキュメント点数) / 15 × 100
```

## 評価実施方法

### フェーズ 1：定量指標計測

1. **各プランの実装完了後**
   - ベンチマークテストを実行
   - コード複雑度を計測
   - カバレッジレポートを生成

2. **メトリクス収集**
   ```bash
   # 全ベンチマーク実行
   uv run pytest local_packages/image-annotator-lib/benchmarks/ -v

   # 複雑度計測
   uv run radon cc -a local_packages/image-annotator-lib/image_annotator_lib/core/

   # カバレッジ計測
   uv run pytest --cov=image_annotator_lib.core local_packages/image-annotator-lib/tests/
   ```

3. **スコア計算**
   - 各指標をスコアに正規化
   - 重み係数を適用
   - 総合スコアを算出

### フェーズ 2：定性指標評価

1. **評価者グループ**
   - 評価者：3～5名（リード開発者）
   - 評価方法：独立実施 → 平均スコア算出

2. **各項目の評価**
   - デバッグ容易性：実際のエラーテスト
   - 設定の簡潔さ：ドキュメント・コード レビュー
   - PydanticAI追従性：API 使用率・コード分析
   - LoRAIro統合影響：統合テスト・ドキュメント確認

3. **スコア算出**
   - 各評価者のスコアを収集
   - 平均値を計算
   - 異なる評価の理由をディスカッション

### フェーズ 3：総合評価

1. **総合スコア計算**
   ```
   総合スコア = Σ(定量指標スコア × 重み) + Σ(定性指標スコア × 重み)
   ```

2. **実装アプローチの評価**
   - 総合スコア比較
   - 指標別の強弱分析
   - メリット・デメリット抽出

3. **推奨アプローチの選定**
   - 総合スコアが高いアプローチを推奨
   - ただし、特定の重要指標で大きく劣る場合は検討

## 附録：計算スクリプト

評価スコア自動計算用 Python スクリプト：

```python
# scripts/evaluate_pydantic_ai_factory.py

def calculate_normalized_score(metric_value, metric_type):
    """メトリクスを正規化スコアに変換"""
    if metric_type == "cyclomatic_complexity":
        return max(0, 100 - (metric_value * 5))
    elif metric_type == "loc":
        return max(30, 100 - (metric_value - 500) / 50) if metric_value > 500 else 100
    elif metric_type == "coverage":
        return metric_value  # 0～100
    elif metric_type == "load_time_ms":
        return max(20, 100 - (metric_value - 50) / 5) if metric_value > 50 else 100
    # ... 他の指標
    return 0

def calculate_total_score(metrics_dict):
    """総合スコアを計算"""
    weights = {
        "cyclomatic_complexity": 0.20,
        "loc": 0.15,
        "coverage": 0.15,
        "load_time_ms": 0.15,
        "inference_time_ms": 0.10,
        "memory_mb": 0.10,
        "provider_addition_cost": 0.15,
    }

    total = 0
    for metric, value in metrics_dict.items():
        if metric in weights:
            score = calculate_normalized_score(value, metric)
            total += score * weights[metric]

    return total
```

実行方法：
```bash
python scripts/evaluate_pydantic_ai_factory.py --plan 1 --output results/plan1_evaluation.json
python scripts/evaluate_pydantic_ai_factory.py --plan 2 --output results/plan2_evaluation.json
```
